# Item CRUD Architecture

> A unified create / read / update / delete design that serves **all 7 item types**
> (snippet, prompt, command, note, file, image, link) through one set of mutations,
> one set of query fetchers, one dynamic route, and shared type-adaptive components.
>
> Generated by `/research item-crud-research.md`. Grounded in the project's existing
> conventions (see `src/actions/profile.ts`, `src/lib/db/items.ts`,
> `src/lib/db/collections.ts`, `src/components/dashboard/*`).
>
> **Source note:** the prompt cited `@docs/content-types.md` and `@src/lib/constants.tsx`.
> Neither exists — the equivalent type reference is `docs/item-types.md`, and the
> type constants live in `src/components/dashboard/type-icons.ts` (+ `TYPE_ORDER` and
> `toLabel` in `src/lib/db/items.ts` / `src/lib/db/helpers.ts`). This design uses the
> real files.

---

## 1. Guiding Principle

There is **one `Item` table** with a `contentType` discriminator (TEXT / FILE / URL)
and an `itemTypeId`. So the data layer and mutations are **type-agnostic** — a single
`createItem` / `updateItem` / `deleteItem` handles every type. The only thing that
varies per type is **how it's presented and which fields the form shows** — and that
variation is pushed entirely into the **components**, not the actions.

> **Rule of thumb:** Actions and fetchers never `switch` on type. Components do.

This mirrors how the codebase already works: `getSidebarItemTypes`, `ItemCard`, and
`CollectionCard` all stay generic and resolve type-specific icon/color via
`getTypeIcon()` + inline `color` styles.

## 2. File Structure

```
src/
├── actions/
│   └── items.ts                 # ALL item mutations (create/update/delete + toggles)
│
├── lib/
│   ├── db/
│   │   ├── helpers.ts           # requireUserId(), toLabel()  (existing)
│   │   └── items.ts             # ALL item queries (existing fetchers + new ones)
│   └── items/
│       ├── schema.ts            # Zod schemas + per-type field config (shared by action & form)
│       └── content-type.ts      # contentTypeFor(typeName) helper (lifted from seed.ts)
│
├── app/
│   └── items/
│       ├── layout.tsx           # reuses AppShell (sidebar + topbar), force-dynamic
│       └── [type]/
│           └── page.tsx         # the ONE dynamic list route for every type
│
└── components/
    └── items/
        ├── ItemListView.tsx     # server: header + grid/list of cards for a type
        ├── ItemCard.tsx         # presentational card (adapts by type)         *
        ├── ItemDrawer.tsx       # client: slide-in create/edit drawer shell
        ├── ItemForm.tsx         # client: form that swaps fields by contentType
        ├── fields/
        │   ├── TextFields.tsx   # content + language  (snippet/prompt/command/note)
        │   ├── FileFields.tsx   # upload + fileName/fileSize  (file/image)
        │   └── UrlFields.tsx    # url input  (link)
        ├── ItemActions.tsx      # client: pin/favorite/delete buttons → call actions
        └── DeleteItemDialog.tsx # client: AlertDialog confirm (mirrors DeleteAccountDialog)

* An item-detail ItemCard already exists at src/components/dashboard/ItemCard.tsx for
  the dashboard's compact rows. The list view can reuse it or introduce a richer
  variant under components/items/. Keep one shared card; don't fork per type.
```

**Why this split** (matches `coding-standards.md` "File Organization"):

- **`actions/items.ts`** — mutations only. Client components and forms call these.
- **`lib/db/items.ts`** — reads only. Server components call these directly.
- **`app/items/[type]/`** — one route, not seven.
- **`components/items/`** — the only place that knows a "file" looks different from a "link".

## 3. Routing: `/items/[type]`

One dynamic segment serves all type-filtered list pages
(`/items/snippets`, `/items/prompts`, … `/items/links`), exactly the `href`s the
sidebar already generates (`/items/${type.name}s` in `getSidebarItemTypes`).

```
src/app/items/[type]/page.tsx
```

```tsx
import { notFound } from "next/navigation";
import { getItemsByType, resolveItemType } from "@/lib/db/items";
import { ItemListView } from "@/components/items/ItemListView";

export const dynamic = "force-dynamic"; // per-request, session-scoped (existing convention)

// /items/snippets → "snippets". The sidebar links use the plural form.
export default async function ItemsByTypePage({
  params,
}: {
  params: Promise<{ type: string }>; // Next 16: params is async
}) {
  const { type } = await params;

  // Validate against the 7 system types; 404 on anything else.
  const itemType = await resolveItemType(type); // strips trailing "s", looks up by name
  if (!itemType) notFound();

  const items = await getItemsByType(itemType.name);
  return <ItemListView itemType={itemType} items={items} />;
}
```

Key routing decisions:

- **Plural ↔ singular:** the URL is plural (`snippets`); the DB `ItemType.name` is
  singular (`snippet`). `resolveItemType` normalizes (strip trailing `s`) and looks the
  type up — returning `null` for unknown names so the page calls `notFound()`.
- **Pro types (`file`, `image`):** the route still renders; the **upload mutation** is
  gated server-side on `isPro` (the page can show an upgrade prompt instead of an
  upload field for free users — a component concern).
- **Protection:** add `/items` to the `proxy.ts` `PROTECTED_PREFIXES` + `matcher`
  (currently `/dashboard`, `/profile`), so unauthenticated users are redirected to
  `/sign-in`.
- **Layout:** `app/items/layout.tsx` reuses `AppShell` (like `profile/layout.tsx`) so
  the list renders inside the sidebar + top-bar chrome.
- **Single-item view / editing** is handled by a **drawer** (per the project's
  "Quick-access drawer" + "slide-in drawer" UX), not a separate `/items/[type]/[id]`
  route — so there's no second dynamic route to build.

## 4. Reads — `lib/db/items.ts`

Extend the existing fetchers (which already export `DashboardItem`, `itemInclude`,
`toDashboardItem`, `getSidebarItemTypes`) with the list + single-item reads. All keep
the existing pattern: `requireUserId()` scoping, shared `itemInclude`, map to a
view-model, no auth logic in components.

```ts
// Resolve a URL segment ("snippets") → the system ItemType, or null.
export async function resolveItemType(segment: string): Promise<DashboardItemType | null>;

// All of the current user's items of one type, newest first.
export async function getItemsByType(typeName: string): Promise<DashboardItem[]>;

// A single item the user owns (for the edit drawer); null if not found/not theirs.
export async function getItemById(id: string): Promise<DashboardItem | null>;
```

- `getItemsByType` filters `where: { userId, itemType: { name } }`, `orderBy`
  `{ isPinned: "desc" }, { createdAt: "desc" }` (pinned float to top, per spec).
- `getItemById` **must** filter by `userId` too — never trust an id alone (ownership =
  authorization, the same rule the profile actions follow).
- Reuse `toDashboardItem` / `itemInclude` so list and dashboard share one shape.

## 5. Mutations — `actions/items.ts`

One file, `"use server"`, following `profile.ts` exactly: Zod validation →
`requireUserId()` → ownership check → Prisma write → `revalidatePath` →
`{ success, error?, data? }`.

```ts
"use server";

import { revalidatePath } from "next/cache";
import { z } from "zod";
import { prisma } from "@/lib/prisma";
import { requireUserId } from "@/lib/db/helpers";
import { itemInputSchema } from "@/lib/items/schema";
import { contentTypeFor } from "@/lib/items/content-type";

export interface ActionResult<T = void> {
  success: boolean;
  data?: T;
  error?: string;
}

export async function createItem(input: ItemInput): Promise<ActionResult<{ id: string }>>;
export async function updateItem(id: string, input: ItemInput): Promise<ActionResult>;
export async function deleteItem(id: string): Promise<ActionResult>;

// Lightweight toggles (optimistic UI in the list/card):
export async function toggleFavorite(id: string): Promise<ActionResult>;
export async function togglePinned(id: string): Promise<ActionResult>;
export async function touchItem(id: string): Promise<ActionResult>; // sets lastUsedAt on open/copy
```

Each mutation:

1. **Validate** input with one Zod schema from `lib/items/schema.ts`
   (`safeParse`, return `parsed.error.issues[0]?.message` like `profile.ts`).
2. **`requireUserId()`** — scope every write to the session user.
3. **Ownership on update/delete/toggle:** `where: { id, userId }` (use
   `updateMany`/`deleteMany` so a mismatched owner affects 0 rows, then check
   `count === 0` → "Not found"). Never `update({ where: { id } })` without the user.
4. **Derive, don't ask:** `contentType` is computed via `contentTypeFor(typeName)` —
   the client never sets it. `itemTypeId` is resolved from the type name server-side.
5. **Pro gate:** in `createItem`/`updateItem`, if the resolved type is `file`/`image`,
   require `user.isPro` (the only type-aware branch in the whole action layer, and it's
   an *authorization* check, not presentation logic).
6. **Revalidate:** `revalidatePath("/items/" + pluralType)` and `/dashboard` so the
   list, sidebar counts, and dashboard refresh.
7. **try/catch** → `console.error("[createItem] failed:", e)` → generic error message.

> File/image **upload** itself (R2, multipart, progress) stays an **API route**
> (`/api/upload`) per `coding-standards.md` ("Use API routes … for file uploads with
> progress tracking"). The form uploads to `/api/upload`, gets back a `fileUrl`, then
> calls `createItem` with that URL. The action stores metadata; it doesn't stream bytes.

## 6. Where Type-Specific Logic Lives (Components)

The actions and fetchers are flat. **All branching by type happens in components**,
driven by `contentType` (TEXT / FILE / URL) and the type's `icon`/`color`.

| Concern | Lives in | How it adapts |
|---|---|---|
| Icon | `type-icons.ts` → `getTypeIcon(icon)` | name → Lucide component (existing) |
| Accent color | inline `style={{ color }}` | from `ItemType.color` (existing pattern) |
| Which form fields show | `ItemForm.tsx` → `fields/*` | switch on `contentType` |
| Card body rendering | `ItemCard.tsx` | TEXT → code/markdown preview + copy; FILE → thumbnail/file row; URL → link |
| Label / plural | `toLabel()` | "snippet" → "Snippet" / "Snippets" (existing) |
| Pro upgrade prompt | `ItemForm.tsx` | `file`/`image` + `!isPro` → show upsell instead of upload |

### `ItemForm.tsx` (the one type-aware switch)

```tsx
// Picks the field set from the resolved contentType — the only place the
// per-type difference is encoded on the write path.
function FieldsFor({ contentType }: { contentType: ContentType }) {
  switch (contentType) {
    case "TEXT": return <TextFields />; // content (+ language for snippet/command)
    case "FILE": return <FileFields />; // upload → /api/upload, fileName/fileSize
    case "URL":  return <UrlFields />;  // url
  }
}
```

Shared fields (`title`, `description`, tags, collection membership) render the same for
every type, above the `FieldsFor` block.

## 7. Component Responsibilities

| Component | Client/Server | Responsibility |
|---|---|---|
| `app/items/[type]/page.tsx` | Server | Resolve type, 404 invalid, fetch items, render `ItemListView`. No UI. |
| `ItemListView.tsx` | Server | Header (type icon/label/count, "New" button) + responsive grid of `ItemCard`s. Empty state. |
| `ItemCard.tsx` | Server (presentational) | Render one item, tinted by type color; embeds `ItemActions`. Adapts body by `contentType`. |
| `ItemActions.tsx` | Client | Pin / favorite / copy / edit / delete buttons; calls the toggle/`touchItem` actions; optimistic UI + toast. |
| `ItemDrawer.tsx` | Client | Slide-in shell (create or edit) wrapping `ItemForm`; opens without leaving the list. |
| `ItemForm.tsx` | Client | Shared fields + `FieldsFor(contentType)`; client validation mirroring the Zod schema; calls `createItem`/`updateItem`. |
| `fields/TextFields` `FileFields` `UrlFields` | Client | The three field sets, one per `contentType`. |
| `DeleteItemDialog.tsx` | Client | `AlertDialog` confirm (mirrors `DeleteAccountDialog.tsx`) → `deleteItem`. |

Server-component-first per `coding-standards.md`: only the interactive pieces
(`'use client'`) are the drawer, form, actions row, and delete dialog. The page, list
view, and card stay server-rendered.

## 8. End-to-End Flows

**Read (list):**
`/items/snippets` → `page.tsx` (server) → `resolveItemType` + `getItemsByType` →
`ItemListView` → `ItemCard[]`. No client JS needed to see the list.

**Create:**
"New" button → `ItemDrawer` opens with `ItemForm` (type preselected from the route) →
user fills shared + type fields → (FILE: upload to `/api/upload` first) → `createItem`
action → validate / scope / derive `contentType` / Pro-gate / write →
`revalidatePath` → drawer closes, list shows the new card, toast.

**Update:**
Edit on a card → `ItemDrawer` in edit mode (`getItemById` preload) → `updateItem(id,…)`
with `where: { id, userId }` ownership → revalidate.

**Delete:**
Delete → `DeleteItemDialog` confirm → `deleteItem(id)` (`deleteMany where { id, userId }`)
→ cascade clears `ItemCollection` / `ItemTag` rows → revalidate.

**Toggle / use:**
Pin/favorite → `togglePinned`/`toggleFavorite` (optimistic). Copy/open → `touchItem`
updates `lastUsedAt` (powers "Recently Used").

## 9. Conventions Carried Over (checklist)

- ✅ `{ success, data?, error? }` result shape on every action (from `profile.ts`).
- ✅ `requireUserId()` for scoping; ownership via `where: { id, userId }`.
- ✅ Zod `safeParse` → first issue message; validate on both client and server.
- ✅ One shared `itemInclude` + `toDashboardItem` view-model (no N+1).
- ✅ `export const dynamic = "force-dynamic"` on item routes (session-scoped reads).
- ✅ Server components by default; `'use client'` only for drawer/form/actions.
- ✅ Type → icon via `getTypeIcon`, accent via inline `color` (existing card pattern).
- ✅ `contentType` derived server-side via `contentTypeFor`, never client-supplied.
- ✅ Pro (`file`/`image`) enforced server-side, not just hidden in the UI.
- ✅ File upload via `/api/upload` API route; action stores only metadata.
- ✅ Add `/items` to `proxy.ts` `PROTECTED_PREFIXES` + `matcher`.

## 10. Summary

| Layer | Count | File(s) | Type-aware? |
|---|---|---|---|
| Mutations | 1 file | `src/actions/items.ts` | No (except Pro authz) |
| Queries | 1 file | `src/lib/db/items.ts` | No |
| Route | 1 dynamic | `src/app/items/[type]/page.tsx` | Only validates the segment |
| Components | shared set | `src/components/items/*` | **Yes** — by `contentType` |

One mutation file, one query file, one route, shared components that adapt by type.
The single `Item` table makes the write/read path uniform; the only per-type branching
lives in the form's field set and the card's body — exactly where presentation belongs.
